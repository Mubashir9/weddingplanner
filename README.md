# Wedding Planner

A shared planning tool built for me and my wife for our wedding.

---

## What it does

Six pages, each solving one problem:

**Dashboard** — the first thing you see. A countdown to the Shaadi, four stat cards (tasks done, budget spent, vendor count, venues confirmed), a per-event task progress panel, and an AI nudge panel that reads your actual data and tells you what needs attention. The nudges are generated by Gemini 2.5 Flash and cached for 6 hours so they do not re-run on every load.

**Events** — dedicated cards for Nikkah, Shaadi, and Valima. Shows the date, venue, and notes for each. All three fields are editable inline — changes save directly to the database.

**Tasks** — a checklist grouped under each of the three events. Click a task's status badge to cycle it through `todo → in progress → done`. Add tasks inline per event, choosing a category and who it is assigned to (Mubashir or Kulsoom). Delete tasks with an inline confirmation step. Both users see the same list in real time.

**Vendors** — a card board for every vendor you are considering or have booked. Stores name, category, event, Instagram link, phone, quote in PKR, status, and notes. Filter by event or category. Cycle the status from shortlisted to contacted to booked. When a vendor is marked booked and has a quote, a matching budget item is automatically created (deduped). Delete vendors with an inline confirmation.

**Budget** — line items grouped by event, plus a general bucket for costs that span all three. Each item has a paid toggle. Running total at the top. Items auto-created from vendor bookings show "via [vendor name]" so you know where they came from.

**AI Chat** — a streaming conversation with Gemini that has your full wedding context loaded into every message. It knows your events, tasks, vendors, and budget. Strictly limited to wedding-related topics. Includes an in-UI help popup explaining what the assistant can and cannot do. It does not browse the internet and does not persist conversations across sessions.

### Key architectural decisions

**Server vs client components.** Pages fetch data server-side (no loading spinners, no client-side Supabase queries leaking into the bundle). Client components — `TasksClient`, `VendorsClient`, `BudgetClient`, `EventsClient`, `ChatInterface` — handle mutations and real-time interaction only.

**AI context injection.** The AI is not agentic. It does not call tools or query the database itself. `lib/wedding-context.ts` serialises the current state of tasks, vendors, budget, and events into a plain object that gets JSON-stringified into the system prompt on every request. Simple, predictable, cheap.

**Nudge caching.** The `/api/nudges` route checks the `ai_nudges` table for a row generated within the last 6 hours before calling Gemini. This means opening the dashboard does not trigger an API call on every visit. You can force a refresh from the dashboard nudge panel.

**Vendor → Budget auto-link.** When a vendor's status is changed to `booked` and they have a quote set, the app checks for an existing `budget_items` row with that `vendor_id`. If none exists, it inserts one automatically. The link is visible on the Budget page as "via [vendor name]" on the row. Un-booking a vendor does not delete the budget item.

**Task assignment.** `assigned_to` is a plain text column (not a user FK). Tasks can be assigned to Mubashir or Kulsoom from a dropdown when creating. The assigned name renders as a pill on the task card.

**AI topic guardrail.** The Gemini system prompt explicitly instructs the model to refuse any question unrelated to this wedding. Off-topic questions receive a fixed redirect message.

**RLS.** Every table has Row Level Security enabled. A helper function `get_user_wedding_id()` returns the caller's `wedding_id` from `wedding_members`, and all policies are written against that. Neither user can read or write rows belonging to a different wedding.

## Database schema

Six tables:

- `weddings` — one row, the wedding record itself
- `wedding_members` — joins users to the wedding, stores role (`owner` / `editor`)
- `events` — the three events with dates and optional venue/notes
- `tasks` — per-event tasks with status, category, and `assigned_to` (text)
- `vendors` — vendor records with category, status, contact info, quote
- `budget_items` — spend line items linked to an event and optionally a vendor
- `ai_nudges` — cached Gemini nudge responses with timestamp

## Running locally

```bash
cd wedding-planner
npm install
npm run dev
```

Open `http://localhost:3000`. You will be redirected to `/login`.

Before the app does anything useful you need Supabase and a Gemini key — see `SETUP.md` in the repo root for the full walkthrough.

## Design

White background. One accent color (`#B5886A`, a muted warm terracotta). Two fonts — Cormorant Garamond for display text (the days counter, event names, page titles) and DM Sans for everything functional. No shadows on cards, borders only. No gradients. No rounded-full on anything except status badges and the assigned-to pill.
